# Feature Specification: 交互动画精修与 image2 分层资产升级

**Feature Branch**: `002-animation-delight-upgrade`

**Created**: 2026-06-24

**Status**: Draft

**Input**: 逐个优化 39 个母题的交互动画，让动画更精致、更有趣、更搞笑；图片资产使用 image2 生成，并且要以大图交互舞台为主，不再把动画退化成题干旁的小图。

## Clarifications

### Session 2026-06-24

- 本次需求先更新 SpecKit 文档，暂不实现动画代码。
- 新建独立分支 `002-animation-delight-upgrade`，后续实现围绕本规格推进。
- image2 图片不是生成整张静态插画，而是按背景、可动对象、状态帧、数学道具、反馈道具和 fallback 分层规划。
- M09、M12、M20、M21、M39 已经方向正确，作为第一批基准模块，只做精修和质量门统一。
- 所有题目参数、过程量、标签和答案继续保持整数，不向孩子展示小数。

## User Scenarios & Testing

### User Story 1 - 孩子看到真正的大舞台动画 (Priority: P1)

孩子打开任意母题后，首先看到题目主题明确的大尺寸交互舞台。动画对象会随着参数、步骤和拖动操作发生真实变化，而不是只在题干旁显示一张小图。

**Why this priority**: 这是用户明确指出的核心问题。没有大舞台和动态对象，图片再精致也只是装饰。

**Independent Test**: 打开任一标记为“已精修”的母题，在桌面和手机视口分别调整参数、切换步骤、复位，确认舞台对象、量线和关系式同步变化。

**Acceptance Scenarios**:

1. **Given** 已精修母题，**When** 用户进入学习页，**Then** 主舞台是首屏最大内容块，题干旁不出现替代主体验的小缩略动画。
2. **Given** 用户拖动或调整一个关键参数，**When** 参数变化，**Then** 舞台内对象数量、位置、长度、面积或分组至少有一项同步变化。
3. **Given** 用户切换步骤，**When** 动画进入下一步，**Then** 量线、标签、关系式和反馈与当前步骤一致。
4. **Given** 用户点击复位，**When** 状态恢复，**Then** 参数、对象位置、反馈和步骤都回到当前子题默认状态。

---

### User Story 2 - image2 资产能服务交互而不是割裂交互 (Priority: P1)

开发者为每个精修母题生成 image2 图片资产时，必须先拆清楚哪些是背景、哪些是可动对象、哪些是状态帧和数学道具。Vue 动画只使用这些图层组装舞台，不把答案或可变数字画死在图片里。

**Why this priority**: 用户已经指出 M39 的问题是“图片和交互割裂”。资产契约不清楚时，后续还会重复出现整图静态化。

**Independent Test**: 对任一精修母题检查 asset manifest，确认每个必需角色都有可加载路径，并且图片内不含题干、公式、答案或可变数字。

**Acceptance Scenarios**:

1. **Given** 一个精修模块，**When** 查看其 asset manifest，**Then** 至少包含 `background`、`actors`、`measureProps`、`feedbackProps` 和 `fallbackStill`。
2. **Given** 一个可变数量对象，例如书本、筹码、积木、方格或柱条，**When** 数量变化，**Then** 前端通过重复渲染或移动独立对象实现，不切换整张静态答案图。
3. **Given** 图片加载失败，**When** 模块进入 fallback，**Then** 仍保留题干、控件、关系式、整数计算和可恢复错误提示。
4. **Given** image2 生成的新资产，**When** 进入仓库，**Then** 文件命名、路径、来源提示词和用途记录完整。

---

### User Story 3 - 好笑反馈帮助孩子理解数学状态 (Priority: P1)

孩子操作时会看到轻松有趣的反馈，例如试商太大顶到上限、余数排队、括号亮灯、车尾提醒还差一点、书本回架排队。这些反馈必须解释当前数学关系，不嘲笑孩子，也不遮挡关键量。

**Why this priority**: 用户希望动画更有趣、更搞笑。幽默必须变成学习反馈，而不是额外装饰。

**Independent Test**: 在 M01、M02、M03、M31 和一个应用题模块中分别触发一次错误或边界状态，确认反馈贴近相关对象、短暂出现、说明原因且不遮挡主要标签。

**Acceptance Scenarios**:

1. **Given** 用户产生可纠正状态，**When** 反馈出现，**Then** 反馈必须绑定当前数学状态，并说明或暗示下一步应调整什么。
2. **Given** 用户开启减少动态效果，**When** 同一反馈触发，**Then** 不使用位移或弹跳，改用静态提示、边框、高亮或文字表达。
3. **Given** 正确完成一步，**When** 系统确认，**Then** 反馈短暂且克制，不使用全屏庆祝。
4. **Given** 反馈出现，**When** 舞台中有关键量线或关系式，**Then** 反馈不得遮挡这些信息。

---

### User Story 4 - 家长和评审能按批次验收动画精修 (Priority: P2)

家长、产品评审和开发者可以按模块批次查看哪些母题已精修、资产是否齐全、交互是否通过验收、哪些问题需要返工。

**Why this priority**: 39 个母题一次性精修风险很高，必须有批次和状态，才能稳定推进。

**Independent Test**: 完成一个批次后，打开验收文档和自动化结果，确认每个模块都有状态、截图记录、资产记录和失败项。

**Acceptance Scenarios**:

1. **Given** 一个模块完成精修，**When** 更新验收记录，**Then** 记录包含模块 ID、资产清单、关键交互、三视口结果、减少动态结果和遗留问题。
2. **Given** 一个模块尚未精修，**When** 在母题库或文档查看状态，**Then** 不应被误标为已通过。
3. **Given** 某模块验收失败，**When** 查看失败记录，**Then** 能定位到断图、遮挡、非整数、反馈不清或舞台过小等具体类别。

## Edge Cases

- image2 资产还未生成时，模块必须保持现有可用体验，不破坏学习路径。
- 某个角色对象数量很大时，舞台允许分组显示，但总数标签必须精确。
- 图片对象过多影响性能时，必须使用分组、虚拟化或合成层优化，不允许改成静态答案图。
- 图片内若误生成文字、公式或数字，不能直接使用，必须重生成或裁切处理。
- 移动端舞台空间不足时，控件下移或折叠，不能把主舞台缩成小卡片。
- 快速连续拖动时，旧动画必须取消并落到最后一次输入状态。
- 减少动态模式不能隐藏学习所需信息。
- 所有除法、平均分、速度和分数相关参数必须自动校正为整数可解状态。

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 为 39 个母题维护独立的动画精修状态，至少包含 `not-started`、`planned`、`in-progress`、`accepted`、`blocked`。
- **FR-002**: 每个已精修母题 MUST 使用大尺寸主舞台承载核心交互，不得退化为题干旁的小图片或纯公式卡片。
- **FR-003**: 每个已精修母题 MUST 至少有一个直接操作能够改变舞台对象状态。
- **FR-004**: 每个已精修母题 MUST 在参数、步骤、题干、关键量、关系式和答案之间保持同一状态源。
- **FR-005**: 每个已精修母题 MUST 保持所有参数、中间量、标签和答案为整数。
- **FR-006**: 每个已精修母题 MUST 支持复位，复位后清除临时反馈并回到当前子题默认状态。
- **FR-007**: 每个已精修母题 MUST 支持 `prefers-reduced-motion`，减少动态后信息仍完整。
- **FR-008**: 每个已精修母题 MUST 至少包含一个与数学状态绑定的轻幽默反馈。
- **FR-009**: 幽默反馈 MUST 不遮挡关键量、关系式、题干和主要控件。
- **FR-010**: 幽默反馈 MUST 不嘲笑孩子，不把错误操作作为羞辱性内容。
- **FR-011**: image2 资产 MUST 按图层角色记录为 `background`、`actors`、`stateFrames`、`measureProps`、`feedbackProps`、`fallbackStill`。
- **FR-012**: image2 图片内 MUST NOT 包含可变数字、题干、公式、答案或会随题目参数变化的文字。
- **FR-013**: 可变数量对象 MUST 通过独立对象重复渲染、移动、分组或程序化遮罩实现，不得使用整张静态答案图。
- **FR-014**: 每个 image2 资产 MUST 记录来源提示词、用途、模块 ID、文件路径、是否透明和是否可重复渲染。
- **FR-015**: 每个已精修母题 MUST 提供图片加载失败 fallback，且 fallback 不影响数学计算和控件使用。
- **FR-016**: 系统 MUST 为 M09、M12、M20、M21、M39 保留现有核心交互方向，并只做精修、资产治理和质量门统一。
- **FR-017**: 系统 MUST 优先用 M01、M02、M03、M31 作为第二阶段样板，验证运算顺序、括号优先、位值进位和组合面积的精修模式。
- **FR-018**: 系统 MUST 按批次推进其余模块：M04-M08、M10-M19、M22-M27、M28-M38。
- **FR-019**: 每个批次完成后 MUST 更新验收文档，记录已通过、失败项和返工项。
- **FR-020**: 系统 MUST 在桌面、平板和手机三视口中保持主舞台无横向溢出、无文本重叠、无控件遮挡。
- **FR-021**: 系统 MUST 对精修模块执行断图检查，所有引用图片 `naturalWidth > 0`。
- **FR-022**: 系统 MUST 对精修模块执行对象数量或状态断言，防止图片存在但交互割裂。
- **FR-023**: 系统 MUST 对精修模块执行快速连续操作测试，最终状态必须与最后一次输入一致。
- **FR-024**: 系统 MUST 避免新增全局重型动画引擎，除非某个模块的复杂时间线无法用 Vue、CSS transition 或 Web Animations API 清晰维护。
- **FR-025**: 系统 MUST 保留现有账号、鉴权、管理员和蓝图加载边界，本次动画精修不得扩大认证范围。
- **FR-026**: 系统 MUST 保留 38 个知识点、39 个母题、117 个子题及其 K/M 关系。
- **FR-027**: 系统 MUST 为每个已精修母题提供家长可读的一句话动画讲解和一个常见错因提示。
- **FR-028**: 系统 MUST 记录每个模块的 image2 生成批次、最终采用资产和废弃原因。
- **FR-029**: 系统 MUST 为动画精修提供可搜索的状态或文档索引，便于评审查看进度。
- **FR-030**: 系统 MUST 在构建或测试中校验 asset manifest、模块注册、整数约束和验收记录的完整性。

### Key Entities

- **AnimationUpgradeRecord**: 某个 M 编号的精修状态、批次、质量门和验收记录。
- **Image2AssetManifest**: 模块的 image2 分层资产清单，记录角色、路径、提示词和可交互用途。
- **SceneStageSpec**: 模块主舞台的布局、图层、对象、控件和响应式规则。
- **PlayfulFeedbackSpec**: 与数学状态绑定的轻幽默反馈定义。
- **IntegerControlSpec**: 只产生整数状态的滑杆、步进器、输入框或拖动控制定义。
- **VisualAcceptanceRecord**: 三视口、减少动态、断图、对象状态和截图验收结果。

## Success Criteria

### Measurable Outcomes

- **SC-001**: M09、M12、M20、M21、M39 在精修后保留现有核心操作成功率 100%，且通过新质量门。
- **SC-002**: M01、M02、M03、M31 四个第二阶段样板均具备 image2 分层资产、动态对象和轻幽默反馈。
- **SC-003**: 每个标记为 `accepted` 的模块均通过 1280x720、834x1112、390x844 三视口无横向溢出检查。
- **SC-004**: 每个标记为 `accepted` 的模块均通过断图检查，所有引用图片 `naturalWidth > 0`。
- **SC-005**: 每个标记为 `accepted` 的模块均有至少一个自动或人工记录的对象状态断言。
- **SC-006**: 每个标记为 `accepted` 的模块在题干、参数、过程量和答案中不出现小数。
- **SC-007**: 每个标记为 `accepted` 的模块在减少动态模式下仍能完成核心学习路径。
- **SC-008**: 每个标记为 `accepted` 的模块至少有一条幽默反馈验收记录，确认其绑定数学状态且不遮挡关键量。
- **SC-009**: 全部 39 个模块完成精修后，母题库中“图片动画”标签和精修状态与 asset manifest 一致率 100%。
- **SC-010**: 全部精修完成后，生产构建、组件测试、核心 E2E 和视觉验收记录全部通过。

## Assumptions

- image2 由 Codex 图像生成能力或后续等价工具提供；本规格只定义产物和接入要求。
- 旧图片资产可以继续作为参考或 fallback，但不再作为割裂交互的整图答案。
- 第一阶段不引入音效，避免家庭学习环境中产生噪声。
- 账号、学习进度、数据库和部署流程保持现状，不作为本特性的主要改造范围。

## Out Of Scope

- 重写全部数学内容、题干和答案规则。
- 新增考试系统、排行榜、AI 批改或学习记录同步。
- 为动画精修引入新的账号角色或公开内容编辑后台。
- 一次性要求所有 image2 图片在文档阶段生成完成。
